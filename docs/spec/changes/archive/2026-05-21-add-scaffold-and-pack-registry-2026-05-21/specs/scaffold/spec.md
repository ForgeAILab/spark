## ADDED Requirements

### Requirement: Template Registry

The system SHALL maintain a registry of base scaffold templates under `templates/<name>/`. Each template directory MUST contain a `template.toml` declaring `name`, `status` (one of `stable` or `planned`), `provides` (an array of **template-capability** tags from the closed template-capability enum — separate from the pack-capability enum), and a one-line `description`. The set of registered templates in v1 MUST be exactly `nextjs`, `astro`, `astro-starlight`, `vite-react`, and `one`. The template-capability enum for v1 is exactly: `static`, `server`, `react`, `native`, `vue`, `svelte`, `mdx-content`, `edge-runtime`. Template-capabilities MUST NOT overlap with pack-capabilities; a tag valid in one enum is not valid in the other.

#### Scenario: All five templates are registered

- **WHEN** the system is built from this change
- **THEN** `templates/nextjs/template.toml`, `templates/astro/template.toml`, `templates/astro-starlight/template.toml`, `templates/vite-react/template.toml`, and `templates/one/template.toml` all exist
- **AND** each parses as a valid `template.toml`

#### Scenario: Only nextjs is stable in v1

- **WHEN** the registry is loaded
- **THEN** `templates/nextjs/template.toml` declares `status = "stable"`
- **AND** the other four templates declare `status = "planned"`

#### Scenario: Expo is not registered in v1

- **WHEN** the registry is loaded
- **THEN** no `templates/expo/` directory exists
- **AND** `expo` is not an accepted value for the `--template` flag of `create-app-skills`

### Requirement: Minimal AI-Ready Scaffold (nextjs reference)

The `create-app-skills` initializer SHALL generate a minimal Next.js 15 + TypeScript application when the user selects the `nextjs` template. The generated app MUST be preloaded with the board-driven AI workflow artifacts (`AGENTS.md`, `CLAUDE.md`, `.ai/`, `.claude/skills/`, `.codex/skills/`) and nothing else. The generated app MUST NOT include auth, database, UI component library, payments, email, or AI SDK integrations by default; those capabilities are added later via the pack registry.

#### Scenario: Fresh init with nextjs and no preset

- **WHEN** the user runs `bunx create-app-skills my-app --template nextjs` and declines any preset
- **THEN** a directory `my-app/` is created containing a runnable Next.js 15 app
- **AND** `my-app/AGENTS.md`, `my-app/CLAUDE.md`, `my-app/.ai/`, `my-app/.claude/skills/`, and `my-app/.codex/skills/` exist
- **AND** `bun dev` starts the dev server successfully without prompting for environment variables
- **AND** no auth, db, UI lib, payments, email, or AI SDK code is present in the generated tree

#### Scenario: Init with a preset

- **WHEN** the user runs `bunx create-app-skills my-app --template nextjs --preset saas-classic`
- **THEN** the base scaffold is generated as in the no-preset case
- **AND** the CLI subsequently invokes `app-skills preset saas-classic` against the new directory
- **AND** the packs declared by that preset are installed before the command returns

### Requirement: Planned Templates Refuse Init

The initializer SHALL refuse to scaffold a project against a template whose `status` is `planned`. The refusal MUST be clear, name the requested template, and direct the user to the follow-up issue or documentation.

#### Scenario: Init against a planned template fails clearly

- **WHEN** the user runs `bunx create-app-skills my-app --template astro` in v1
- **THEN** the CLI exits non-zero
- **AND** the error message names `astro` and says it is `planned, not yet implemented`
- **AND** no directory `my-app/` is created

#### Scenario: Interactive picker marks planned templates

- **WHEN** the user runs `bunx create-app-skills my-app` with no `--template` flag
- **THEN** the interactive picker lists all four registered templates
- **AND** the three planned templates are visually marked (e.g. with a `(planned)` suffix)
- **AND** selecting a planned option produces the same refusal as the `--template` flag form

### Requirement: Scaffold Carries Canonical Skills

The base scaffold copied by `create-app-skills` SHALL include the canonical set of board-workflow skills under `.claude/skills/`, and a mirrored copy under `.codex/skills/` generated by the project-wide skill mirror. Pack-shipped skills are added later when packs install, not at scaffold time.

#### Scenario: Skills are present after init

- **WHEN** the scaffold finishes
- **THEN** `.claude/skills/` contains at minimum `mvp-grill`, `mvp-spec`, `architecture-cutline`, `mvp-board`, `board-review`, `implementation-brief`, `execute-task`, `code-review`, `qa-verify`, `sync-board`, `next-task`, `risk-check`, plus the three new skills `pack-resolve`, `pack-add`, `new-pack`
- **AND** every skill present in `.claude/skills/` has a corresponding entry in `.codex/skills/`

### Requirement: Scaffold Records Config

The scaffold SHALL write `app-skills.config.json` at the repository root capturing the app name, the chosen template name, and any template variables required by future pack installs (e.g. database URL placeholders, port numbers).

#### Scenario: Config file is present and valid

- **WHEN** `create-app-skills my-app --template nextjs` finishes
- **THEN** `my-app/app-skills.config.json` exists
- **AND** the file is valid JSON
- **AND** the file contains `appName: "my-app"` and `template: "nextjs"`
