## The full pipeline you should run

The first phase should be **interactive grilling**, not building. This is the part most teams skip, and it is exactly the part Lovable formalizes. Start with a rough app idea, but do not let the planner output a PRD immediately. Force it to ask only the questions that change scope or architecture: who the user is, the single most important user action, the first use case that must work, what can be faked manually, where the data comes from, whether auth is required, what ŌĆ£doneŌĆØ means for a first release, and what absolutely must not be built yet. LovableŌĆÖs docs repeatedly stress foundation-first prompting, breaking work into ŌĆ£bricks,ŌĆØ and using planning before code; GitHubŌĆÖs Spec Kit similarly inserts a clarification step before technical planning. The second phase is **specification**. Do not jump from chat directly into tasks. Create one concise artifact that becomes the source of truth for the MVP. AnthropicŌĆÖs long-running harness research also found that agents need explicit feature lists and handoff artifacts to avoid either trying to build everything at once or declaring the job done too early. The third phase is **editable technical planning**. This is where you mimic Lovable most directly. ## The board should be the cockpit

If you want the whole system to work **with or without Forge**, make the boardŌĆönot the chat threadŌĆöthe primary control surface. GitHub Projects is the most practical default because it already supports table, kanban board, and roadmap views, plus custom metadata, issue/sub-issue relationships, and tight connection to pull requests. A good minimal board has a few fields that matter more than status churn: **Epic**, **Task type**, **Owner agent**, **Human owner**, **Status**, **Validation state**, **Risk**, **Linked PR**, and **Demo URL**.
